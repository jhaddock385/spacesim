## Design Phase

Reviewed and discussed the full game concept: a Star Trek-style bridge sim where the player issues natural language commands to AI crew members (Claude API), who interpret those commands and act against a deterministic ship simulation. Three-layer architecture: simulation (pure Lua), AI agents (Claude API), interface (Love2D rendering).

Key design decisions made:

- **Simulation model: real-time with freeze.** Not turn-based. The simulation runs continuously; it freezes when the player issues commands or on periodic auto-freeze intervals. Real-time chosen because it's easier to tune physics by feel — watch the ship move, adjust thruster values, iterate.
- **Fixed timestep physics (60Hz).** Simulation ticks at a fixed rate regardless of frame rate, with rendering interpolating between states.
- **Event bus as the simulation backbone.** All state changes flow through a central event bus at every scale — within subsystems (engineering components), between subsystems (tactical to shields), and between entities (ship to ship). This replaced an earlier narrower proposal (`combat.lua`) after discussion about needing the bus pattern everywhere, including internal engineering problems (reversing polarities, rerouting plasma through conduits).
- **V1 scope cuts.** Impulse only (no warp), phasers only (no torpedoes), single shield value, simple power pool, 3 crew members. Enough to prove the concept.
- **Dev display first, player UI later.** Full visibility into the simulation for development. What the player eventually sees is a separate question answered through playtesting.
- **Tool use for AI responses.** Claude API function calling instead of text parsing for crew actions. Fallback to speech-only on malformed output.
- **Cascade limits.** Max depth on event chains and AI agent cascades to prevent runaway loops.

## Implementation: Core Foundation

Built the simulation foundation — event bus, state store, spatial physics, ship subsystems, and dev display. Everything flows through the bus; nothing directly mutates another system's state.

**Core infrastructure (`src/core/`):**
- `bus.lua` — event bus: emit, subscribe with filtering, drain with depth limit (20), wildcard subscriptions, event history for debug display
- `store.lua` — partitioned state store keyed by entityId/component/key
- `clock.lua` — fixed timestep accumulator (60Hz), freeze/unfreeze with reason tracking, time scale

**Simulation layer (`src/sim/`):**
- `entities.lua` — entity registry with auto-incrementing string IDs, create/destroy lifecycle events
- `spatial.lua` — 2D physics: position, velocity, rotation with damping (0.995 velocity, 0.98 rotation), thrust and torque event handlers
- `ship/init.lua` — ship factory wiring spatial + engineering
- `ship/engineering.lua` — warp core generates discrete power pips (2/sec, pool of 10), player allocates pips to thrusters via bus events
- `ship/helm.lua` — reads thruster pip allocations, emits thrust/torque force events (50 force/pip, 2 torque/pip)

**Dev display (`src/screens/simulator.lua`):**
- Spatial map with grid, ship triangle, velocity vector, camera follow + scroll zoom
- System panel: warp core status, pip pool visualization, per-thruster allocation with +/- buttons
- Event log: scrolling color-coded display of all bus events

**Full event chain working:** Click +button -> bus emit -> engineering allocates pip -> helm reads power -> emits thrust event -> spatial applies force -> position updates -> ship moves on screen.

## AI Crew System

Built the full captain-to-crew pipeline. The player types a natural language command, it goes through prompt construction, the Claude API with tool use, and actions execute against the simulation via the bus.

**Station definitions** (`src/data/stations.lua`): Data-driven description of what each station can see (store paths), do (actions as Claude API tools), and query. Started with helm station — allocate/deallocate thrusters + navigation readout.

**Crew knowledge** (`src/data/knowledge/helm.lua`): Prose descriptions of ship systems written for AI consumption. Explains thruster physics, power pip tradeoffs, heading system, and role expectations. Structured to eventually support competency levels (level 1 vs level 5 knowledge).

**Prompt builder** (`src/agents/prompt.lua`): Assembles the full system prompt from: identity (rank + name + role), behavioral framing (rules for how crew should act), working knowledge, and a live state snapshot from the store. Also builds Claude API tool definitions from station actions, and resolves tool calls back to bus events.

**Text input** (`src/ui/textinput.lua`): Typeable text field with cursor, basic editing, Enter to submit. Used for captain commands in comms mode.

**Comms mode in simulator**: Toggle between "Direct Control" (manual +/- buttons) and "Comms" (AI crew) via a visible button. In comms mode, typed commands go through the full pipeline. Dialogue log shows captain commands, crew speech, and system actions.

**Tool use continuation loop**: When the API returns `stop_reason: tool_use`, the game executes the tools (actions mutate state via bus, queries read from store), sends results back, and lets the API continue generating. This enables crew members to check instruments before responding verbally.

**Async API calls**: Solved the macOS beach ball problem. Instead of blocking `io.popen`, curl runs in the background writing to a temp file. The update loop polls each frame. The app stays fully responsive during API calls — ship keeps moving, UI stays interactive.

**Conversation history**: Multi-turn context within a session. Kim remembers previous commands and can respond to follow-ups like "yes" referencing a prior suggestion.

**Results from first playtesting:**
- "Full speed ahead" → Kim allocates all 10 pips to main thruster. Correct.
- "Half speed" → Kim deallocates 5 pips. Correct math.
- "Report on heading and status" → Kim queries navigation, then responds with a detailed verbal report including position, heading, speed, and power allocation.
- "Increase forward speed by 2 and port thruster by 1" → Kim allocates exactly 2 main + 1 port. Correct.
- "Can you change heading to northward?" → Kim explains the current thruster situation, suggests swapping port for starboard, asks for confirmation. When told "yes", executes the swap. Good reasoning, correct actions.
